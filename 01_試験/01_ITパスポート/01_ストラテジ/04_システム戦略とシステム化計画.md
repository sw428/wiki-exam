# システム戦略とシステム化計画

## 目的

このファイルは、ITパスポートのシステム戦略系問題を、問題文の段階と手がかり語で切れるように整理する場所です。

情報システム戦略、システム化構想、システム化計画、要件定義は、全部「システムを考える話」に見えます。試験では、どの段階で何を決めているかを見ます。

## 情報システム戦略

### 何の問題か

情報システム戦略では、ITだけで勝手に方針を立てるのではなく、会社の経営戦略と合っているかを見ます。

### 小学生レベルの言い換え

情報システム戦略は、会社の作戦に合うITの作戦を立てることです。

会社が「ネット販売を強くしたい」と言っているなら、IT側もそれを支えるシステムや仕組みを考えます。

### 問題文の手がかり

- 情報システム戦略
- 経営戦略
- 整合性
- 会社の目的
- 事業方針

### 選択肢の切り方

- 経営戦略との整合性を図るなら、情報システム戦略の中心
- 開発期間の短縮は、開発現場やプロジェクト管理寄り
- コスト削減は大事だが、戦略の必須中心ではない
- 最新技術の導入は、会社の目的に合うかが先

### 最後に見る一言

情報システム戦略 = ITの作戦を会社の作戦に合わせる。

## システム化計画プロセス

### 何の問題か

システム化計画プロセスでは、業務を実現するためのシステム機能の範囲と内容を定義します。

すでに「システム化する方向」は見えていて、次に「どの業務をどこまでシステムにするか」を決める段階です。

### 小学生レベルの言い換え

システム化計画は、「どこまでをシステムにするか」を決める段階です。

店の仕事で考えると、注文、在庫管理、会計、配送の全部をシステム化するのか、在庫管理だけをシステム化するのかを決めます。

### 構想・計画・要件定義の住み分け

| 段階 | 見るもの | 一瞬フレーズ |
| --- | --- | --- |
| システム化構想 | 問題点、課題、なぜシステム化するか | なぜやる |
| システム化計画 | 機能の範囲、内容、どこまでシステム化するか | どこまでやる |
| 要件定義 | 利用者や業務に必要なこと | 何が必要 |
| 設計・開発 | 画面、詳細設計、作る | 作る |

### 選択肢の切り方

- `機能の範囲`、`内容を定義` ならシステム化計画
- `問題点を分析`、`課題を定義` ならシステム化構想寄り
- `経営上の課題`、`ニーズ` なら情報システム戦略や構想の前段階寄り
- `画面の詳細` なら、もっと後の設計工程

### 誤答しやすい理由

「システム化する」という言葉だけを見ると、全部同じ段階に見えます。

でも試験では、次のように段階で切ります。

```text
構想 = なぜやる？
計画 = どこまでやる？
要件定義 = 何が必要？
開発 = 作る
```

## 調達プロセスとRFP

### 何の問題か

調達プロセスでは、システムを外部のベンダに依頼するために、RFPを作り、調達先を選び、提案書を評価します。

試験では、誰が何をするかと、順番を見ます。

### 小学生レベルの言い換え

ユーザ側を先生、ベンダ側を生徒として考えます。

ユーザは、先に採点表を作ってから、ベンダに課題を説明します。ベンダは答案として提案書を作り、ユーザがそれを採点します。

### 基本の流れ

```text
要件仕様書作成
-> RFP作成
-> 調達先選定基準作成
-> 調達先候補の選定
-> RFP説明
-> 提案書作成
-> 提案書評価
-> 調達先選定
-> 契約締結
```

### 問題文の手がかり

- RFP
- 調達先
- 選定基準
- 調達先候補
- 提案書
- 提案書評価
- 契約締結

### 選択肢の切り方

- `調達先選定基準作成` は、先に採点表を作る段階
- `RFP説明` は、ユーザが候補ベンダへ課題を説明する段階
- `提案書作成` は、ベンダが回答として提案を書く段階
- `提案書評価` は、ユーザが選定基準に沿って提案を採点する段階

### 誤答しやすい理由

「選定基準」は、提案書を見た後に作るものに見えます。

でも調達では、先に基準を作っておかないと、あとから都合よく採点しているようになります。だから順番は、基準を作ってから説明、提案、評価です。

## 最後に見る一言

情報システム戦略は経営戦略に合わせる。システム化計画は機能の範囲と内容を決める。調達は、基準 -> 説明 -> 提案 -> 評価。
