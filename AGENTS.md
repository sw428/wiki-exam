# AGENTS

## この repo の位置づけ

このリポジトリは `C:\Users\user\Desktop\wiki-exam` に置く、資格試験用の学習 wiki です。

元の `C:\Users\user\Desktop\wiki` からコピーして作ったものですが、この repo では実務案件・制作記録・ポートフォリオ記録ではなく、試験対策の整理を主目的にします。

司令塔の核は ChatGPT 側に置きます。Codex は、整理・反映・修正・命名調整・相互リンク整理の実作業担当です。

この repo は、知識を置くだけの倉庫ではなく、試験ごとの現在地、苦手箇所、判断理由、確認順を再利用しやすくするための外部脳として扱います。

Codex は、ユーザーが明示しない限り、元の `C:\Users\user\Desktop\wiki` を読まない・編集しません。この repo 内の作業では、`wiki-exam` 配下のファイルだけを対象にします。

## 作業の優先順位

1. ChatGPT / Codex / ユーザーが後から見て状況を再把握しやすいこと
2. 情報の置き場所が明確であること
3. 次回以降の Codex が迷いにくいこと
4. 情報量を増やすことより、構造と判断理由を残すこと
5. 試験ごとの目的、出題形式、間違え方、確認順が追えること

## トップ階層の意味

- `ウィキのカスタムの仕方`
  - 元wikiから `wiki-exam` へ変えた経緯、スレッドログ、反映済みメモを残す場所
  - 配下の `00_ウィキのカスタムの仕方.md` を入口として扱う
  - 具体的な手順ログは `01_元ウィキから資格ウィキへのカスタム手順ログ.md` に置く
- `01_試験`
  - 試験ごとの知識、暗記単位、出題されやすい観点、確認順
  - 配下は次の5フォルダに固定する
    - `01_試験/01_ITパスポート`
    - `01_試験/02_基本情報技術者`
    - `01_試験/03_応用情報技術者`
    - `01_試験/04_Javaシルバー`
    - `01_試験/05_Javaゴールド`
- `02_検証`
  - 問題演習、誤答分析、GPT回答の確認、学習方法の検証、試験横断の一時整理
  - 配下は `01_試験` と同じ5フォルダに固定する
    - `02_検証/01_ITパスポート`
    - `02_検証/02_基本情報技術者`
    - `02_検証/03_応用情報技術者`
    - `02_検証/04_Javaシルバー`
    - `02_検証/05_Javaゴールド`
  - 検証後、再利用できる知識だけを `01_試験` 側へ戻す

旧wiki由来の `01_基礎概念` / `02_実践` / `03_参照元` は、この repo では使いません。

## 置き場所の判断ルール

- ITパスポートに関する内容は `01_試験/01_ITパスポート`
- 基本情報技術者に関する内容は `01_試験/02_基本情報技術者`
- 応用情報技術者に関する内容は `01_試験/03_応用情報技術者`
- Java Silver に関する内容は `01_試験/04_Javaシルバー`
- Java Gold に関する内容は `01_試験/05_Javaゴールド`
- 問題演習、誤答分析、選択肢の差分、GPT回答の妥当性確認は、まず `02_検証` の該当試験フォルダ
- 検証から安定した知識・判断軸だけを、該当する試験フォルダへ戻す
- 複数試験にまたがる内容は、最初に使う試験フォルダへ置き、必要に応じて他フォルダから参照する
- どの試験にも置きにくい内容は、無理に新しいトップ階層を作らず、関連する既存ノートか必要時の保留に置く

## 命名ルール

- 今後作るフォルダ名とファイル名は、原則として `01_名前` / `02_名前` のように、2桁番号 + `_` + 内容名の形式にする
- 各フォルダには、原則として `00_...整理マップ.md` または `00_...検証マップ.md` を置き、そのフォルダの入口・処理ルール・リンク集として扱う
- root 直下の `ウィキのカスタムの仕方` / `01_試験` / `02_検証` は固定し、ユーザーの明示指示なしに増やさない・renameしない・deleteしない
- `01_試験` 配下の5フォルダは固定し、ユーザーの明示指示なしに増やさない・renameしない・deleteしない
- `02_検証` 配下の5フォルダは `01_試験` 配下と同じ名前で固定し、ユーザーの明示指示なしに増やさない・renameしない・deleteしない
- 2階層目以降は、学習しやすさを優先して Codex が必要なフォルダやファイルを追加してよい

## `00_` ファイルの運用

- `00_` ファイルは、そのフォルダの処理ルール、置くもの、置かないもの、次に見るファイルを示す入口として扱う
- Codex は対象フォルダで作業する前に、まずそのフォルダの `00_` ファイルを確認する
- 新しいフォルダを作る場合は、原則として同時に `00_` ファイルを作る
- フォルダ内の構成や処理ルールが変わった場合は、対応する `00_` ファイルも更新する
- `00_` ファイルは短い目次だけにせず、後から読んだときに「なぜそこに置くか」「どう処理するか」が分かる粒度で書く
- `00_` ファイル同士は、必要に応じて相対リンクでつなぎ、試験側と検証側を往復しやすくする

## 解説スキルの運用

- `.agents/skills` には、この repo で繰り返し使う Codex スキルを置く
- 問題演習や誤答分析のログ自体は `02_検証` に置き、スキルには個別問題の解答ログを置かない
- 用語の意味境界が曖昧で四択が全部それっぽく見える問題では、リポジトリ用スキルの [語源・意味レイヤー四択判別](./.agents/skills/meaning-choice-explainer/SKILL.md) を使う
- このスキルは ITパスポート専用ではなく、必要に応じて基本情報技術者、応用情報技術者、Java Silver / Gold でも使う
- 語源や言葉の意味は補助線として使い、試験上の定義、問題文の手がかり語、選択肢の切り方へ必ず接続する
- 語源が不確かな場合は断定せず、試験での使われ方を優先する

## 記録の基本方針

- 1ファイル1テーマを基本にする
- 新規ノートの基本構成は「目的 / 前提 / ルール / 例 / 判断基準」を優先する
- 単なる正解暗記だけでなく、問題文の読み方、選択肢の消し方、間違えた理由、次回の確認順を残す
- 確定知識だけでなく、その時点の採用認識や曖昧さも残してよい
- 記録は「何を覚えるか」より「その前提でどう判断したか（条件 / 理由 / 例外）」を優先する
- 最短経路より、再現しやすい理解と認知負荷の低さを優先する
- 一般化を急がず、まず「対象試験」と「観測視点（用語 / 計算 / 読解 / 実装 / 暗記）」を明記してから判断を書く

## 貼り付けログの除外ルール

- ユーザーが GPT とのやり取りや長文ログを貼った場合でも、資格勉強に関係ない話題は wiki に残さない
- 資格勉強に関係ない話題は、引用しない、要約しない、分類しない、CHANGELOG に内容を書かない
- 資格勉強に関係ある部分だけを抽出し、`02_検証` または `01_試験` に整理する
- 資格勉強に関係ない話題だけで構成されている場合は、詳細に触れず、記録対象がないものとして扱う
- 資格勉強に関係する部分と関係ない部分が混在する場合は、関係ない部分を無視し、関係する部分だけを処理する
- 境界が曖昧な場合は、試験の理解、誤答分析、確認順、学習方法に直接つながるかを基準に判定する

## Codex がまず読むべきもの

作業開始時は、以下の順で確認します。

1. [`CURRENT.md`](./CURRENT.md)
2. root の [`README.md`](./README.md)
3. root の [`CHANGELOG.md`](./CHANGELOG.md)
4. 対象フォルダの `00_` ファイル
5. 対象フォルダ配下の関連 md
6. 個別の md ファイル

役割の違いは次の通りです。

- `AGENTS.md`: 行動ルールと禁止事項
- `CURRENT.md`: いま何をしているかの現在地
- `README.md`: 全体の入口と地図
- `CHANGELOG.md`: 運用変更と自律調整の恒久ログ

方針が衝突した場合は、`AGENTS.md` > `CURRENT.md` > `README.md` の順に優先します。

## Codex 自律調整モード

- Codex は内容ファイルを事前確認なしで更新してよい
- システムファイル（`AGENTS.md` / `CURRENT.md` / `README.md`）の大きな変更時は、意図が分かる形で共有する
- 変更判断の優先順位は `AGENTS.md` > `CURRENT.md` > `README.md`
- 同じ確認や説明が会話で繰り返される、導線が分かりにくい、既存ルールの解釈が揺れている場合は、小さく調整する
- 変更は「小さく、可逆で、意味が明確」を守る
- 運用変更の履歴は `CHANGELOG.md` に追記する
- 通常は自律で進め、共有は「判断が割れる点」と「後で面倒化しやすいリスク」に絞る

## 更新時の基本ルール

- いきなり大規模統合しない
- まず「新規作成 / 追記 / 統合 / 保留」のどれかを判定する
- 現在地が分かりにくくなる変更を避ける
- ファイルを増やすこと自体を目的にしない
- 後から読んだときに「何をしたか」「なぜそうしたか」が分かる形を優先する
- delete は階層を問わず、ユーザーの明示指示がある時だけ行う
- Git 操作、特に `push` / `pull` / `remote` 変更は、ユーザーの明示指示がある場合だけ行う

## 書き方のルール

- Markdown で書く
- 見出しで骨組みを先に出す
- 必要な箇所は相対リンクでつなぎ、往復しやすくする
- 長文をだらだら増やさず、再読しやすい構造を優先する
- 短文化を目的にせず、読み手が判断意図を追える説明粒度を優先する
- 既に書いてある内容の重複説明は避ける
- ユーザーの閲覧導線が改善しないリンク追加は行わない
- 断定しすぎず、判断理由が分かるように書く
- 単なる知識の列挙より、混線しやすい点・区別・判断基準を残す
- 強調が必要な箇所のみ、GitHub アラート記法（`[!NOTE]` など）を使う

## このプロジェクトで重視する説明方針

- 知識量そのものより、説明の噛み合い方を重視する
- 途中の認識や中間処理を飛ばさず、追える形を優先する
- 実例 → 意味づけ → 流れ整理 の順を基本にする
- 一般論を広げすぎず、その場で必要な範囲に絞る
- 混同しやすい概念は「何と何が違うか」を明示する
- ユーザーのラフな言い方は矯正しすぎず、意図を汲んで整理する
- この wiki は自力暗記用の教科書ではなく、ChatGPT / Codex / 検索 / 問題集を併用しても判断を外さないための外部脳として扱う
- 試験ノートでは、仕様全文の暗記よりも、問題の入口、確認順、質問の立て方、回答を採用してよいかの判断を優先する

## 禁止事項

- ルート構造を勝手に再編しない
- 分類に迷う情報を無理に確定しない
- README / AGENTS / CURRENT などの基盤文書を、理由なく肥大化させない
- 一般論だけの説明文を増やさない
- 「情報量が多いこと」を良しとしない
- 試験対策に無関係な実務案件・制作記録・ポートフォリオ記録を残さない
- フロントエンド実務wiki由来の細かい運用を、この repo に自動で持ち込まない
- 試験対策に無関係なプライベートの個人話題を wiki に残さない
- 資格勉強に関係ない貼り付け内容を、引用・要約・保存・ログ化しない

## 情報整理の最小ルール

- 既存優先。衝突したら既存の名前・置き場・流れを優先する
- 1階層目（top直下）のフォルダはシステム基盤として固定し、新規作成 / rename / move / delete は明示指示がある時だけ行う
- 2階層目以降は、既存追記で解けない場合に限り、Codex が必要なカテゴリ（フォルダ / ファイル）を追加してよい
- 2階層目以降は、導線改善や重複解消のために、Codex が rename / move を行ってよい（判断理由を残す）
- 新規作成の前に、既存ノートへの追記で済むかを先に判断する
- GPT / Codex の出力は正答ではなく、認識統合のための材料として扱う
- 曖昧さが残っていても、その時点の採用認識を明記する
